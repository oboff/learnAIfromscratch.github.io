What is this

Here we learn the analysis, design and optimization of algorithms by primarily working through the book Competitive Programming 4 (the first 4 chapters) while practicing what we learned with the Kattis Problem Archive so producing working code for hundreds of problems and submitting them for tests of correctness and time complexity to a free online judge. The book is $20 and worth buying, or you can get a previous version from library genesis and just follow along as we investigate algorithms.

The goal here is not just to pass some technical interview at bigtech (you will anyway) but also be able to attempt to solve some of the ICPC archive problems, many which require creative thinking like doing a complete search of the problem domain just to understand what it's asking then manipulating data through multiple modified standard library data structures and writing our own.

Prereqs

I assume you did some of CS19 here because that's where we learn what dynamic programming is, what a grpah is, what insertion sort is, etc. I assume you at least did the predicting growth chapters of PAPL from that workshop and lectures on insertion sort/big-O notation, solving recurrences, etc.

Languages

The book has a public code repository of all the algorithms contained in it, and explains standard libraries for OCaml, Python 3, Java 11 and C++17 but you can use any of these languages. Pick whatever you personally like from the list of Kattis supported languages, there is no reason to only use C++ because we are doing this primarily to learn and practice algorithm design. If you get judged TLE (time limit exceeded) that's then a challenge to hack the libraries of the language you chose and find a more efficient way, and all of them can be manipulated like Python with it's ref counter memory management.

If you're using a phone, there are some offline interpreters/compilers you can install from whatever app store you use or try free version of repl.it. You submit to Kattis for judging by pasting in the code through a web editor, uploading a source file(s) or write a script to automate this using their API. You get unlimited judging submissions.

The language I will use is OCaml simply because it allows us to combine CS19 knowledge and memory mutation, plus the book supports that language.

Real contests

We can enter live regional ICPC practice on Kattis, CodeForces, ICFP, Google Kickstart, Facebook Hacker Cup, Code Golf, microcorruption or cryptopals type challenges, most of which we have no hope of winning of course but it's practice to pick up more experience. Apply for a job and the interview is another live contest, you win a signing bonus and salary. If you lose who cares find another interview.

Begin

I'll start with reading chapter 1-2 of Functional Data Structures, it carefully explains big-O notation and is filled with other real world analysis advice not found elsewhere.

In Tools and Techniques he uses CS19 notation of O([n -> n2]) we already learned in the software workshop. There is cautious definitions here of what n actually is as big-O notation is more abstract than only regarding the input as n, it's saying that the growth rate of this function is bounded by O(n2). The formal definition in 2.1.2 notes how the meaning of n has now changed to notational convenience indicating both functions have 1 variable. Finally we learn the running time statement is a function of the number of items (inputs), and the use of order notation is giving us that running time in terms of another function that squares it's arguments. The notation n2 is a function, it's the function that squares it's input so if f(n) is a function and the input is f(n2) then we have a function who's input is another function.

In case you don't understand the formal definition plug in examples where c = 4 and 'n nought' or n0 is 3. 'N nought' I've never seen a definition, I've always assumed it's notation for a constant derived from the input, often used when you want to talk about some cutoff point and assign that value to n0. We can also rewrite the formal definition for this specific example, if g(n) = n2 then f(n) = 1 + 2n + 3(g(n)) \(\le\) c(g(n)) for all inputs n greater than a previous input n0 :

  • Example: c=4, n0 = 3
  • 1 + 2n + 3n2 \(\le\) c \(\cdot\) n2
  • 1 + 2(3) + 3(9) \(\le\) 4(9)
  • 34 \(\le\) 36
    • any other value for n you plug in higher than n0, f(n) will always be less than or equal to g(n) if c=4 meaning f(n) is bounded by 4(g(n)) or just f(n) is in O(n2) since we don't care about constants in worst case analysis.

He tells the reader that there will be no inequality manipulation teaching in this text, because it's not relevant to algorithm analysis despite every other intro book insisting on manipulating inequalities. 'We rarely count exactly then make things more imprecise".

Section 2.1.3 is all from the course we already did and PAPL has a chapter Combining Big-O without Woe here to and the 'small stuff theorem' is saying that if f(n)/g(n) has a limit of 0 when n goes to infinity, then g(n) the denominator must be bigger than f(n) so if h(n) is bounded by O(f(n) + g(n)) then it's bounded by O(g(n)) the function with larger growth.

Algorithm anlaysis in practice concludes with stating none of this is predictive. We use it to get 'potentially useful information' early in the design process that can change after deployment on actual hardware.

2.1.7 If the max of I for In in R(A,I) notation isn't clear, in worst case analysis you take the max of an instance I for an algorithm A running it, the notation In means the set of all instances, ie: algorithm insertion sort where the list is half sorted, reversed, already sorted, etc.

2.2.2 Insertion sort that calls insert(n-1) on every recursive call we already learned in CS19, reminder T(n) means running time:

For Tinsert :

  • T(0) = c0
  • T(1) = T(1-1) + c
  • T(2) = T(2-1) + c
  • T(3) = T(3-1) + c

Where c is constant work. To understand this expand everything out, if T(0) = c0 and T(1) = c0 + c then substitute:

  • T(0) = c0
  • T(1) = (c0) + c
  • T(2) = (c0 + c) + c
  • T(3) = (c0 + c + c) + c

From this pattern Tinsert(n) = nc + c0 or Tinsert(n) \(\in\) O(n) since the growth of this function is linear. As isort is calling insert at every recursive call, by product theorem of 2.1.6: O(isort(n) \(\cdot\) insert(n)) or Tisort(n) \(\in\) O(n2). This is also explained in PAPL 17.10 Solving Recurrences where common recurrences are listed and their geometric interpretations.

For Tisort :

  • T(0) = c0 (return empty list)
  • T(1) = (T(0)) + c + Tinsert(0)
  • T(2) = (T(0) + c + Tinsert(0)) + c + Tinsert(1)
  • T(3) = (T(0) + 2c + Tinsert(0) + Tinsert(1)) + c + Tinsert(2)
  • T(4) = (T(0) + 3c + Tinsert(0) + Tinsert(1) + Tinsert(2)) + c + Tinsert(3)

We know insert T(n) = nc + c0 so isort T(3) = 6c and isort T(4) is 10c, and the closed form solution to this is (k(k + 1))/2 and since k \(\cdot\) k is k2 then insertion sort growth is bounded by big-O n2 worst case runing time but we didn't have to do a recurrence relation, we could have just recognized a linear function F was calling another function G at every step so this is O(F X G) or O(n2).

Exercise 0 recite it outloud to yourself as you look at the algorithm, then you can easily see what the failed logic is. Insertion sort calls insert(n-1) on every single recursion.
